## 文档说明

这份文档**不讲技术细节**，只讲业务。目的是给非技术人员看，留作历史档案记录"合并方案当时是怎么想的"。

里面会出现一些 `/api/xxx` 这样的"接口地址"。**你不需要理解它是什么**，只需要把它当成一个"业务功能的编号牌"——三个原项目里，每个业务功能都挂了这么一块编号牌，合并的时候要决定哪些编号牌可以撕掉、哪些必须保留。

整份文档分成三个部分：

1. **三个原项目当前都有哪些接口**：把现在 GitHub 上三个仓库里店长和后台用到的每一个接口都列出来，用大白话说清楚每个接口干什么。这是"现状盘点"。

2. **哪些接口是重复或几乎重复的**：找出来"同一件事情在两三个项目里都做了一遍"的情况，给出合并建议。这是"合并方案"。

3. **合并后的统一接口清单**：把合并后的最终方案完整列出来。这是"目标状态"。

**前端尽量不动的原则**：
- 三个原项目的界面长什么样、店长怎么点、点完出现什么页面——这些都不动
- 接口地址（`/api/xxx`）能不改就不改
- 只有那种"做的是同一件事但当前是两套实现"的接口，才需要合并
- 合并的时候每次都会明确写：**这个改动会让哪个项目的前端要小幅调整哪几个地方**

# 第一部分：现状盘点 — 三个项目当前的接口

## 1. 货盘选品（skuSelection）

这个项目共 **35 个接口**。按业务功能分成 13 组。

### 1.1 登录与账号

| 编号 | 业务功能 | 输入 | 输出 | 使用场景 |
|---|---|---|---|---|
| SK-A1 | **店长登录** | 账号、密码 | 登录凭证、用户身份（含所属门店） | 店长在登录页输入账号密码 |
| SK-A2 | **查询账号** | 账号 | 账号详情（含所属门店） | 后台管理列出账号 |
| SK-A3 | **创建门店账号** | 门店编号、门店名称、初始密码 | 新创建的账号 | 后台批量开通门店时使用 |

### 1.2 门店主数据

| 编号 | 业务功能 | 输入 | 输出 | 使用场景 |
|---|---|---|---|---|
| SK-B1 | **查询门店列表 / 单个门店** | 可选：门店编号 | 门店编号、门店名称、地址 | 后台查门店、店长查自己的店 |
| SK-B2 | **新增/更新门店** | 门店编号、门店名称 | 保存后的门店 | 后台开通新店 |

### 1.3 商品与销售数据

| 编号 | 业务功能 | 输入 | 输出 | 使用场景 |
|---|---|---|---|---|
| SK-C1 | **查询当前门店的所有在售商品** | 门店编号 | 商品列表（每项含：编码、名称、品牌、规格、单位、保质期、分类树、批发价、近 30 / 90 天销量销售额、库存等） | 店长进入选品页时拉取 |
| SK-C2 | **批量导入门店商品数据** | 门店编号 + 一批商品记录 | 导入结果统计 | 系统定期把 ERP 推送的销售数据同步进来 |
| SK-C3 | **查询竞品（按大类）** | 大类编码 | 竞品列表（名称、规格、渠道、最新价） | 选品分析时显示竞品对比 |
| SK-C4 | **查询有促销的 SKU 编码** | （无） | 一批 SKU 编码 | 选品时筛选"有促销文案的商品" |
| SK-C5 | **查询促销详情** | 可选：大类 | 按"促销组 → SKU → 促销文案"的层级结构 | 虚拟货架里给商品贴促销牌 |
| SK-C6 | **查询基准 SKU 名单** | （无） | 一批 SKU 编码 | AI 推荐时区分"基础款"和"创新款" |

### 1.4 货架配置

| 编号 | 业务功能 | 输入 | 输出 | 使用场景 |
|---|---|---|---|---|
| SK-D1 | **查询门店所有货架配置** | 门店编号 | 货架列表（每架：组名、宽度、层数、支持品类、显示顺序） | 进入选品工作页时加载 |
| SK-D2 | **新增一个货架配置** | 完整的货架信息 | 保存后的货架 | 后台或店长添加新货架 |
| SK-D3 | **更新一个货架配置** | 门店编号、货架编号、要修改的字段 | 更新后的货架 | 比如改宽度、改层数 |
| SK-D4 | **删除一批货架** | 门店编号、要删的货架编号列表 | 删除结果 | 批量调整门店货架 |
| SK-D5 | **整店替换货架配置** | 门店编号、新的完整配置 | 替换结果 | 后台一次性重新规划整家店 |
| SK-D6 | **查询场景定义** | （无） | 全部场景（场景编号、名称、包含哪些品类） | 选品首页展示所有场景 |

### 1.5 场景调改

### 1.6 货架运行时（拍照、检测、当前状态）

## 2. 价盘管理（priceChange）

这个项目共 **4 个接口**（外加一个直接调用 AI 平台的调用，不算自己后端的接口）。极其精简。

> **重要现状**：当前没有登录功能，门店编号写死在后端代码里（`粤32826`）。所有接口都默认查这一家店。

### 2.1 商品价盘

| 编号 | 业务功能 | 输入 | 输出 | 使用场景 |
|---|---|---|---|---|
| PR-A1 | **查询当前门店在售商品** | （无） | 商品列表（编码、名称、品牌、规格、批发价、当前价、原价、近 30 天销量、是否已调过价） | 价盘工作页加载 |
| PR-A2 | **查询所有 SKU 的价格曲线** | （无） | 按 SKU 编码索引的字典：每个 SKU 有若干"价格周期"（起止日期、价格、月销量、月毛利） | 点开商品详情，画价格走势图 |
| PR-A3 | **查询所有 SKU 的竞品价格** | （无） | 按 SKU 编码索引的字典：每个 SKU 有若干渠道（罗森、7-11、天猫等）的最新价 | 商品详情的"竞品对标"页签 |
| PR-A4 | **提交一次调价** | SKU 编码、原价、新价、操作人 | 保存结果 | 店长在调价弹窗点击"应用"后调用 |

### 2.2 AI 诊断（直连第三方 Dify，未经过自己后端）

| 编号 | 业务功能 | 输入 | 输出 | 使用场景 |
|---|---|---|---|---|
| PR-B1 | **AI 价盘诊断** | 单个 SKU 的当前价、批发价、销量、毛利、竞品价等组合 | AI 给出的诊断文字 + 建议（涨价 / 降价 / 保持） + 来源标记 | 点击"刷新 AI 建议"时批量调用 |

## 3. 活动海报（poster）

这个项目共 **32 个接口**。功能上比另外两个项目更"复杂"，涵盖店长侧 + 完整的超管后台。

### 3.1 登录与账号

| 编号 | 业务功能 | 输入 | 输出 | 使用场景 |
|---|---|---|---|---|
| PO-A1 | **登录** | 邮箱、密码 | 登录凭证 + 用户身份 | 登录页 |
| PO-A2 | **记录登录事件** | 当前门店编号（可选） | 写入回执 | 登录后立即调用，用于审计 |
| PO-A3 | **查询当前用户角色** | （无） | 角色列表（超管 / 店主） | 决定要不要显示后台入口 |

### 3.2 设备与门店绑定

> 这套机制只有海报项目有。原因是：海报项目没有"门店登录"概念，是邮箱登录后再让用户告诉系统"我现在在哪家店"。

| 编号 | 业务功能 | 输入 | 输出 | 使用场景 |
|---|---|---|---|---|
| PO-B1 | **查询当前设备绑定的门店** | 设备编号（浏览器唯一标识） | 已绑定的门店编号或空 | 登录后第一时间查询 |
| PO-B2 | **绑定当前设备到某门店** | 设备编号、门店编号 | 绑定结果 | 首次登录时弹窗让用户选门店 |

### 3.3 海报生成（单张同步）

| 编号 | 业务功能 | 输入 | 输出 | 使用场景 |
|---|---|---|---|---|
| PO-C1 | **生成一张海报（同步）** | 拍的照片、文案、模板编号、可选：自定义风格描述、生成模式（拍照合成 / 仅背景 + 商品官方图 / 多商品混排）、可选商品官方图地址、SKU、品类 | 海报图片地址 + 用到的 AI 模型 + 用到的提示词 | 店长选完商品和文案点"生成"，等十几秒拿结果 |

### 3.4 海报生成（批量队列）

| 编号 | 业务功能 | 输入 | 输出 | 使用场景 |
|---|---|---|---|---|
| PO-D1 | **批量入队** | 1-10 张海报的参数（每张：照片、文案、模板、模式等） | 批次编号 + 每张海报对应的任务编号 | 店长一次选了多个商品要生成 |
| PO-D2 | **认领并处理某个任务** | 任务编号 | 是否认领成功 + 处理结果（已完成 / 处理中 / 出错） | 前端拉取一个待处理的任务，调一次 AI |
| PO-D3 | **列出我最近的活跃任务** | （无） | 最近 2 小时的所有任务 + 当前状态 | 前端轮询，刷新进度 |

### 合并 #1：店长登录

| | 选品 SK-A1 | 海报 PO-A1 |
|---|---|---|
| 用什么登录 | 账号（门店编号） + 密码 | 邮箱 + 密码 |
| 谁能登录 | 门店账户、超管 | 店主、超管 |
| 登录后能拿到什么 | 用户 + 默认门店 + 是否超管 | 用户 + 角色（额外接口）|

**结论：必须合并，但合并方案不只是"把两套接口合一"，而是"统一改成飞书登录 + 兼容老账号"。**

合并方案：
- **新登录方式**：飞书扫码 / 飞书 H5 免登（按 [feishu-auth-integration.md](./feishu-auth-integration.md) 的协议）—— 这是最终目标
- **过渡期兼容**：保留账号 + 密码登录一段时间。账号既支持选品的"粤XXXXX"，也支持海报的邮箱
- **新增"查询当前登录用户"接口**：把"我是谁、能进什么模块、默认在哪家店"一次性返回。原选品和海报项目都没有这个独立接口（选品塞在登录响应里、海报单独走 `getMyRole`），统一后改成一个独立接口
- **价盘项目**：从"无登录硬编码门店"改为"必须登录"。这是必要的代价

**对前端的影响**：
- 选品前端：登录页保留，按"飞书登录优先 + 账号密码兜底"改造（约 2 个页面的修改）
- 海报前端：同上
- 价盘前端：要新增登录页（这是必要的工作）

### 合并 #2：门店列表

| | 选品 SK-B1 | 海报 PO-B1 |
|---|---|---|
| 接口干什么 | 直接列出门店主数据 | 查询"当前设备绑定到哪家店" |
| 字段丰富度 | 门店编号、门店名称、地址 | 仅门店编号字符串 |

**结论：不是"同一件事"——选品的是"门店主数据库"，海报的是"我这台浏览器选的门店"。但都涉及"门店"概念，需要协调。**

合并方案：
- **门店主数据**：以选品的 SK-B1 为基础，补充城市、省份、经纬度等扩展字段，作为唯一的门店列表来源
- **"当前用户能管哪些门店"**：新增接口（原三个项目都没有），返回该用户有权限的门店列表
- **"当前设备绑哪家店"**：保留海报的 PO-B1 和 PO-B2，但绑定的目标对象指向同一份门店主数据
- 海报项目原来只存了一个"门店编号字符串"，合并后这个字符串必须匹配到门店主数据里的真实编号

**对前端的影响**：

### 合并 #3：当前门店的商品 / SKU 数据

这是合并清单里**最重要的一条**。

| | 选品 SK-C1 | 价盘 PR-A1 |
|---|---|---|
| 接口干什么 | 拉门店所有商品 + 完整销售数据 | 拉门店所有商品 + 价格信息 |
| 字段丰富度 | 编码、名称、品牌、规格、单位、保质期、3 级分类、批发价、近 30 / 90 天销量销售额、库存、最后配货日 | 编码、名称、品牌、规格、批发价、当前价、原价、近 30 天销量、是否已调价 |
| 数据来源 | 一组共享的"商品主数据 + 销售事实"表 | **同一组**共享的表 |

**结论：必须合并。两个接口其实是同一份底层数据的两个"取景角度"。**

合并方案：
- **统一为一个接口**：返回字段以选品的 SK-C1 为基础（最全），额外补两个价盘前端需要的字段：「当前价」和「是否已调过价」

- **选品前端**：完全不动（字段是它的超集）

**对前端的影响**：
- 价盘前端：1 处接口替换（接口列表里把 `/api/products` 改为新地址）+ 几个字段名映射

# 附录：合并后对各项目前端的改动清单

这一节专门给前端工程师审核用，列出"合并方案会让前端改哪些地方"。**不涉及业务功能改动，只是接口地址或字段名的调整。**

## 选品（skuSelection）前端改动

| 改动点 | 原接口 / 调用 | 新接口 / 调用 | 改动量 |
|---|---|---|---|
| 登录页 | 账号 + 密码登录 | 加一个"飞书登录"按钮，账号密码作为备选 | 1 个页面 |
| 写操作日志 | `POST /api/logs` | 新的"写审计事件"接口 | 几处埋点 |
| 文件上传 | 自有 OSS 上传 | 统一上传接口 | 通常 0-1 处（如果原本就是封装好的） |
| 其余所有业务接口 | 不变 | 不变 | 0 |

## 价盘（priceChange）前端改动

| 改动点 | 原接口 / 调用 | 新接口 / 调用 | 改动量 |
|---|---|---|---|
| **新增登录页** | 无 | 飞书登录 + 账号密码兜底 | 整页新增 |
| 查商品 | `GET /api/products` | 新的"查询门店在售 SKU"接口 + 字段映射 | 1 处 |
| 查竞品 | `GET /api/competitors` | 新的"查询竞品"接口 + 参数微调 | 1 处 |
| 调价 | `POST /api/adjust-price` | 接口保留，但需要带上当前登录用户 | 1 处 |
| AI 诊断 | 直连 `dify.rollingai.cn` | 改走自己后端的 AI 网关 | 1 处 |
| 查价格曲线 | 不变 | 不变 | 0 |

> 价盘的改动量看起来最大，但主要是因为它现状本来就最简单（没有登录、硬编码门店）。合并是一次性补齐它本就应该有的能力。

## 海报（poster）前端改动

| 改动点 | 原接口 / 调用 | 新接口 / 调用 | 改动量 |

## 总结

合并后的接口总数约 **80 个**，比三个项目接口数相加（35 + 4 + 32 = 71）多出一些，主要是新增了：
- 飞书登录相关的几个接口
- "查询当前用户身份"和"切换门店"等门户必需接口
- 跨业务的"审计事件查询"

**改动量评估**：
- 选品前端：影响最小，主要是登录页改造 + 几处日志埋点替换
- 价盘前端：补齐登录、几处接口替换
- 海报前端：登录页加飞书按钮、选门店弹窗改下拉

**当前进度**：合并方案已批准，骨架已落地（M0），按里程碑分批实现业务逻辑。要追踪具体接口什么时候真正可用，看 [里程碑文档](../milestones.md) 的 M1-M5 拆分。
